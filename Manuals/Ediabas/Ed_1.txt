1. Einführung in BEST/1
BEST heißt BEschreibungssprache für STeuergeräte. Diese Beschreibungssprache soll im
Rahmen des Projekts EDIABAS eingesetzt werden, um die Forderung nach einer
applikationsunabhängigen Beschreibungsmöglichkeit zu erfüllen. Zur Programmlaufzeit werden
Beschreibungsdateien geladen und interpretiert, in denen das Wissen über die Steuergeräte, wie
Adressen und Umrechnungen, verborgen ist (encapsuling). Die Beschreibungssprache BEST/1
stellt also Mittel und Wege zur Verfügung um die in den Steuergeräten abgelegten Daten in
steuergeräteunabhängige Werte wie z.B. Drehzahl oder Gerätenummer zu wandeln. Dabei
werden nur die wirklich von der Applikation benötigten Daten weitergeleitet (information
hiding).
Diese Sprache sollte auch unabhängig vom jeweiligen Hostsystem sein und deswegen eine
maschinenunabhängige eigene Verarbeitung aufweisen. Aus diesem Grunde bietet sich ein
virtueller Prozessor an. Dieser virtuelle Prozessor (im folgenden VP genannt) arbeitet wie ein
richtiger Prozessor eine Folge von einfachen Befehlen (Opcodes) ab. Durch dieses Konzept ist es
möglich höhere Sprachebenen zu definieren, die dann mit Hilfe eines Compilers auf diese
Prozessorebene gebracht werden kann. Damit kann diese höhere Sprachebene im Rahmen des
möglichen Compilerbaus problemorientiert und nicht sprachorientiert geplant werden.
Diese vorliegende BEST/1 Beschreibung enthält genau den Befehlsumfang des virtuellen
Prozessors. Für jeden Prozessorbefehl (OPCODE), der ja eigentlich nur durch eine
Bitkombination dargestellt wird, ist ein kurzes Schlüsselwort (Mnemonic) definiert. Die dem
Befehl übergebenen Parameter, sind Namen von Registern oder Konstanten. Diese Sprache ist
also ein genaues Abbild der im virtuellen Prozessor ablaufenden Befehle und seiner Architektur.

3. Einführung in die Prozessorarchitektur
Der Adressraum des VP ist eine Objektdatei, die die Opcodes in binärer Form enthält. Durch
Absetzen eines Jobs aus der API-Schnittstelle wird dieser Prozessor auf eine bestimmte Adresse
in diesem Adressraum gesetzt und die Abarbeitung gestartet. Es wird sequentiell ein Opcode
nach dem anderen abgearbeitet, bis ein Befehl zum Abbruch, oder ein Fehler erfolgt. Die
Verwaltung der Aufrufe, übernimmt ein übergeordnetes Laufzeitsystem, das direkt von der API
gesteuert wird.
Der VP kennt keinerlei Speicherzugriffe, deshalb konnten Zeiger und Arrayzugriffe vermieden
werden, was wesentlich zu einem sicheren Betrieb beiträgt. Der VP verfügt über 2 verschiedene
Registersätze, deren Größe den Anforderungen angepaßt werden kann. Es gibt zum einen den
universal-Registersatz, der für arithmtische Operationen benützt wird und einen
Stringregistersatz, der das Arbeiten mit Strings und Telegrammen ermöglicht.

6. Darstellung eines Opcodes im OBJ Format
In der Beschreibungsdatei werden Opcodes in einem fest definierten Format abgelegt. Es wurde
eine gleiche Länge aller Befehlsrümpfe vereinbart. Ein Befehlsrumpf besteht immer aus zwei
Bytes. Das erste Byte codiert den eigentlichen Befehl, das zweite Byte enthält die
Adressierungsarten.
BEFEHLSCODE DADR SADR
4 Bit Quelladressierung
4 Bit Zieladressierung
8 Bit Opcode
Bild 4: Opcodedarstellung
An den Befehlsrumpf werden dann die entsprechenden Datenbytes angehängt (ohne Alignment).
Zuerst folgen die Daten der Zieladressierung und darauf die Daten der Quelladressierung. Nach
dem letzten Byte, d.h am Ende des gesammten Opcodes, folgt sofort der nächste Opocde!

Tabelle 1: Alle möglichen Adressierungsarten
Es gibt 16 Arten der Adressierung. Bei 4 Bit sind somit alle Codiermöglichkeiten ausgereizt. Die
Adressierungsart 0 heißt, daß kein Operand angegeben wird. Bei den Adressierungsarten 1 bis 4
wird ein Register angegeben. Die Adressierungsarten 5 bis 8 definieren immediate Operanden.
Die Länge des Operanden, und damit auch die Adressierungsart wird durch ein Suffix, d.h. durch
einen mit einem Punkt abgetrennten Buchstaben bestimmt. Fehlt dieser wird von der
Adressierungsart 6 (Integer) ausgegangen. Mit 9 bis B wird ein Element aus einem Stringregister
indiziert, und mit C bis F werden n-Elemente aus einem Stringregister ab Position m adressiert.
Für jeden Opcode ist eine genaue Liste festgelegt, auf welche Zieladressierung, welche
Quelladressierung folgen darf. Jede andere Kombination führt zu einer Fehlermeldung.

Tabelle 2: Berechnung von Effektiver Adresse und Operationslänge
Die Berechnung der effektiven Adresse ist sehr einfach, weil es nur Register Adressen gibt. Bei
einigen Adressierungsarten muß jedoch noch ein Offset hinzugerechnet werden. Bei den
immediate Adressierungsarten, wird ein Hilfspeicher dazu benutzt um eine sinnvolle effektive
Adresse zu generieren. Die Operationslänge wird im allgemeinen durch die Zieladressierungsart
bestimmt. Bei einigen Kombinationen wird die durch die Zieladresse vorgegebene
Operationslänge durch die Quelladressierung überschrieben. Nähreres ist aus oben stehender
Tabelle zu entnehmen. Die Länge der Stringregsiter ergibt sich aus dem Byte mit dem höchsten
Index, das beschrieben wurde. Mit CLEAR, SERASE oder mit SCUT kann die Länge wieder
verringert werden.

{"HALLO",0.b} String "HALLO" mit 0 abgeschlossen

#'A' #$41.b #65.b Byte 'A' = $41 = 65

Beispiel eines Sourcefiles
Diese Beispieldatei definiert zwei Jobs, INITIALISIERUNG" und "IDENTIFIKATION". Die
Jobnamen stehen in der ersten Spalte und enden mit einem Gartenzaun '#' (1,11). Diese Jobs
können dann aus der Applikation heraus aufgerufen werden. Eine Zeile kann einen Kommentar
enthalten, der dann nicht berücksichtigt wird (2,3). Ein Kommentar beginnt mit einem ';' und
endet am Ende der Zeile. Leere Zeilen werden nicht berücksichtigt (10).
Im allgemeinen haben alle Zeilen das folgende Format:
{ { {Label:} OPCODE { Zieladresse {,Quelladresse} } } {;Kommentar} }
oder
{ COMPILER CODE Datalist }
Labels sind Sprungmarken, auf die sich alle Sprungbefehle beziehen können. Der Befehl "JNZ
id_erg" (17) überspringt also die nächsten Befehl und setzt bei "id_erg:" (19) wieder auf, wenn
das Zeroflag nicht gesetzt ist.
Nur wenn eine Zieladresse angegeben ist, kann es eine Quelladresse geben. Es gibt auch Befehle,
die weder Quell- noch Zieladresse besitzten (4,5,9).
Am Beginn des Jobs INITIALISIERUNG wird eine Parameterliste in das Register S0 geladen
(2). Der Treiber wird geöffnet (4). Die Kommunikationsparameter werden mit den vorher
geladenen Werten eingestellt (5). Im Fehlerfall werden 5 Wiederholungen gemacht (6). Das
Register i0 wird mit 1 geladen (7). Dieser Wert wird dann in (8) als Ergebnis "DONE"
zurückgegeben. Dieser Job ist beendet bei (9).
Am Begin des Jobs IDENTIFIKATION wird das Register S1 mit dem Wert "FEHLER\0" geladen
(12). Der Antwortpuffer S0 wird gelöscht (13). Die Antwortlänge wird auf 5 zu erwartende
Blöcke eingestellt (14) und das Telegramm "03,00,00" an das Steuergerät gschickt (15). Im
dritten Byte der Antwort (von 0 an gezählt) muß dann das Kontrollbyte 00 stehen, um als
korrektes Empfangstelegramm identifiziert werden zu können (16). Wenn dies so ist, wurde beim
Vergleich das Zeroflag gesetzt, und die Abarbeitung wird nach (17) einfach fortgesetzt. Dabei

11. Tabellenbearbeitung mit BEST/1
Es können Stringtabellen definiert werden. Auf diese Kann dann mit den Tabellenbefehlen
zugegriffen werden. Eine Tabelle hat folgende Syntax:
TBEG "tabellenname"
HEAD "spaltenname1","spaltenname2",...,"spaltennamen"
LINE "element1,1","element1,2",...,"element1,n"
LINE "element2,1","element2,2",...,"element2,n"
...
LINE "elementm,1","elementm,2",...,"elementm,n"
TEND
Mit dem Befehl 'TABSET "tabellename"' wird die Tabellenbearbeitung für diese Tabelle
initialisiert. Danach kann mit dem Befehl 'TABSEEK "element2,1","spaltenname1"' z.B.: in der
Spalte 1 nach dem Element "element2,1" gesucht werden. Daraufhin kann mann sich mit den
Befehl 'TABGET s0,"spaltennamex" jedes beliebige Element der Zeile ansprechen, in der das
gesuchte Element vorhanden war.

Einstellen der Kommunikationsparameter
Assembler Syntax:
XSETPAR <ea>
Operation
Einstellen der Kommunikationsparameter <Ziel>
Parameterliste: INT Konzept: BMW-Konzept 1 = 1
BMW-Konzept 2 = 2
BMW-Konzept IHK = 3
BMW-Konzept DDE = 4
BMW-Konzept DS1 = 5
BMW-Konzept DS2 = 6
INT Baudrate
INT Reizadresse
INT Wakeupzeit in 1ms
(0 ist kein Wakeup)
INT Idletime in 1ms
(Zeit zwischen Wakeup und Telegramm)
INT Timeouttime in 1ms
(Zeit in der das SG antworten muß)
INT Regenerationtime in 1ms
(Zeit zwischen den Telegrammen)
INT Telegramendtime in 1ms
(Wartezeit nach dem letzten Byte, nach der auf
Telegrammenden entschieden wird)
Optional:
Bytezwischenzeit
Checksum
Angabe wie die Telegrammchecksumme gebildet und
überprüft werden soll. Wird hier nichts angegeben, wird die
Checksumme automatisch vom Interface gebildet und die
Checksumme des empfangenen Telegramms überprüft. Die
Checksumme ist bitcodiert.
Bit 0 = 0: Die Checksumme des zu sendenden Telegramms
muß vom Benutzer gebildet werden.
Bit 0 = 1: Die Checksumme des zu sendenden Telegramms
wird vom Interface gebildet.
Bit 1 = 0: Die Checksumme des empfangenen Telegramms
wird vom Interface überprüft.
Bit 1 = 1: Die Checksumme des empfangenen Telegramms
wird vom Interface nicht überprüft.

Defaultwerte für den Telegrammvorspann:

Länge -2 52 -1 1 1
Offset 0 0 0 0 0
Flags
-
Adressierungsarten
Ziel Quelle
S -
{} -
Beschreibung
Mit diesem Befehl werden die für die Kommunikation mit einem Steuergerät erforderlichen
Kommunikationsparameter gesetzt. Die Parameter in <Ziel> haben ein interfaceunabhängiges

eingestellten Konzept der Telegrammvorspann mit Defaultwerten belegt.

Seite 189
Y2BCD
Bytefolge in BCD-String konvertieren
Assembler Syntax:
Y2BCD <ea>,<ea>
Operation
Für jeweils ein Byte von <Quelle> 2 Zeichen in <Ziel> ablegen:
Bit 0-3 (Quelle) -> 1. Zeichen (Ziel)
Bit 4-7 (Quelle) -> 2. Zeichen (Ziel)
Flags
-
Adressierungsarten
Ziel Quelle
S S {} S[#]# S[#]R S[R]# S[R]R
Beschreibung
Konvertiert eine Bytefolge in einen nullterminierten String (BCD-codiert). Das S Register wird
vorher gelöscht. Für jedes nichtkonvertierbare Byte werden die Zeichen "**" abgelegt und die
Konvertierung fortgesetzt.
